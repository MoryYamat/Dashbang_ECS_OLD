## 🧠 `ECSManager`の生インターフェースがシステム層に直接露出している問題（2025/08/04）

### 💡話題・気づき
- 各`System`が直接，Entityを管理する`ECSMgr(World)`クラスに依存している
    - `ECSMgr`の仕様が変更になった場合，その影響が再現なく拡大する可能性がある
    - `FSMQuery`のようにインターフェースを設けて**変更の影響を最小化**する仕組みが必要(`FSMQuery`は`ECSMgr`の機能をまとめただけの`QueryObject`と言える)

### 試したこと・考えたこと
- `ECSMgr`クラスの機能を**大きな**共通インターフェース`IWorldEntity`にまとめる
    - メリット
        - DI設計(依存性注入)になじむ
        - ECSの入れ替えが可能になる
    - デメリット
        - 変更の影響を抑えるという目的を果たせない可能性が大きい
        - インターフェースが肥大化する
- `ECSMgr`クラスの機能を**ドメイン・機能単位**のインタフェース(例：`MovementQuery`)にまとめる
    - メリット
        - 関心事に分離され，変更の波及が最小限
        - 必要に応じて拡張できる
        - 意味のある関数名が増えSystemの意図がコードで伝わりやすい
    - デメリット
        - クラス数が増えてしまう
        - 共通化したくなった時に整理が面倒

- 今回のプロジェクトでは`意味`や`目的``責務`によって明確な意味構造を作りたいので**ドメイン・機能単位**でインターフェースを作る方針にする
    - `LayerFeature`層で各ドメインの`Query`を作成してそれをシステムに渡す構造にする
        - `Query`は状態を持たないので複数`Layer`で同時に同じ`Query`を別々に生成しても問題ない

## 🧠 `FSM`の遷移リクエストの`priority`制御（2025/08/05）

### 背景・文脈
- 現在のFSM設計では，状態遷移の評価はすべて定義ドリブンで行われており，1ステップにつき最大1つの遷移が適用される
- `priority`フィールドを持たせているが，その利用は限定的であり，基本的に固定値を想定されている
- 将来的にはFSMスケーラビリティ・制御性向上のために`priority`による遷移制御をどのように活用できるかを検討する

### 🎯 目的
- 同時に複数の遷移条件が成立する状況で，**より意図に沿った遷移を選択**できるようにする
- 外部条件(例：キャラ状態，プレイヤー入力，ほかFSMの状態)に基づき，**動的に優先順位を調整**できるようにする
- 汎用的なFSM制御機構としての抽象度を高める


### ✅ 現時点での方針

- `priority` はまだ定数として運用（動的な評価ロジックは未導入）
- `Completed` または `Interrupted` から `None` への遷移は、**明示的に記述**することで制御している（priority = 0）

### 💡 将来的な拡張案（優先度制御）

#### A. 静的 priority による制御 **(現在はこちら)**
- 定義上の `priority` を比較し、最も高いものだけを採用
- 簡潔かつわかりやすいが、状況依存の制御には向かない

#### B. 動的 priority の導入
- `ISkillFSMCondition` の評価と同時に、**priority を計算する関数**を追加
- 例：
  ```cpp
  struct DynamicPriority {
    int evaluatePriority(const SkillFSMContext& ctx, const SkillDef& def) const;
  };


### 🗒️ メモ
#### 💫`priority`は夢の塊
- **`priority`は遷移選択の「評価関数」となる**
    - $priority = f(context, transition)$　という評価関数を設計していくことで，多様で知的なシステムを実現できる
