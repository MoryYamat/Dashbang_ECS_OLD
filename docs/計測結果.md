100体
- [SUM 60 frames] fixedSteps=20
- Movement calls=2020
- Skill calls=2020

ここから逆算すると：
- calls_per_axis = 2020
- calls_per_axis / fixedSteps = 2020 / 20 = 101

FSM Class = 2020 + 2020 = 4040

1000体
- [SUM 60 frames] fixedSteps=240
- Movement calls=240240
- Skill calls=240240
- FSM calls=480480

逆算すると：  
240240 / 240 = 1001  
FSM calls = 240240×2 = 480480  


### 安定区間同士で比べると（100体: 4040、1000体: 480480）：  
calls比 = 480480 / 4040 = 118.93倍  
内訳は： 
Entities比 ≈ 1001 / 101 ≈ 9.91倍（だいたい10倍）  
fixedSteps比 = 240 / 20 = 12倍  
Axes比 = 2 / 2 = 1倍  
掛け算： 9.91 × 12 ≈ 118.9倍 ✅（ぴったり）  
つまり “118倍” は 完全に（エンティティ10倍）×（fixedSteps12倍）。

100体  
代表行: `calls=2040 avg=1.37us total=2801.80us avg=1.37us`
- 60フレームで合計 2801.8 = 2.80ms / 60frames
- 1フレーム当たり: 2.80ms / 60 = 0.0467 ms /frames


1000体  
代表行: `calls=240480 total=342893.6us avg=1.43us`
- 60フレームで合計 342893.6 µs = 342.9 ms / 60 frames
- 1フレームあたり：342.9ms ÷ 60 = 5.72 ms / frame

100体->1000体で、エンティティ数は10倍
- `calls` は 2040 -> 240480 で 約118倍 (callsが100倍になる理由は、`fixedSteps`が12倍回っているので、Entities x fixedSteps となっている)
  - `calls ≈ Entities × fixedSteps`
- `total` は 2.8ms -> 323 で 約122倍

`avg`は1.3~1.4μs でほぼ一定で、処理1回コストは一定  
つまり、「回数が増えた分だけtotalが増える」= 線形スケール  

もし、`movement`以外の直交FSMとして追加で増えていけば、線形に増えていく

FSM更新コストは  
**$O(Entities \times ActiveAxes \times UdateFrequency)$**  
でスケールする
各 AxisInstance::tickの平均コストは一定であり  
軸数・エンティティ数に対して線形に増加することを確認した.


改善方法として考えられるのは、  
- 変化がない場合`tick`しないこと
- 「イベントがあったエンティティだけ更新」すること
- クールダウン中/停止中は更新しない

有効軸数を減らすこと
- 不要な軸はつけない
- 今動く可能性がある軸だけ有効化する

CostperTickを下げる(細かい最適化)
