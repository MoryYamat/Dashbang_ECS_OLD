# 描画システムのリファクタリングに関するメモ

> 2026-01-21



## リファクタリングの目的

大規模リファクタリングの理由(課題)
- 描画システムが全体の拡張されたアーキテクチャ設計とうまく協調できない部分が出てきたため
- 定常的な`3Dモデルの描画`だけでなく、VFXなど瞬間的な`Sprite/Particle`の描画に対応する必要が出てきたため


リファクタリングによって達成する目標
- データ駆動で一括定義できるようにする
- 実リソースから実行系の識別子(ID)を生成(現在はモデルデータComponentをフィルタにしているため巨大)
- ID解決をO(1)で.(これは現在の`Audio`などのほかのリソースと同じ)
- 上のアーキテクチャをベースに変更が容易な設計と実装を目指す



## 設計

まず、描画システム系の主機能は、  
**ウィンドウスクリーンに表示されるすべてのリソースの描画を管理・更新すること**

次に、ゲームにおける描画データを以下のように大別する

### ゲームロジックから見た分類
- アクター(ゲーム内に存在する意味を作るオブジェクト)
- VFX(アクターの動きや相互作用を修飾し演出を強化するエフェクト)
- UI(プレイヤーに情報を提示する描画)
- デバッグ(開発者用機能)



#### アクター
アクターはアニメーションの有無によって以下のように分けられる
- スケルタルメッシュ(ボーンと連動してアニメーションによって動きに変化が存在するモデル)
- 静的メッシュ (地形メッシュなど動きは基本的にないモデル)
- インスタンスメッシュ(草木や石など、同一メッシュを大量の異なる座標に描画する)
- VertexAnimated(VATなどシェーダー側で頂点変位を制御するもの)

描画システムから見れば、アクターは  
「どの定数バッファ(行列)を使い、どのシェーダーバリアントを呼び出すか」の集合体  



#### VFX
VFXはエフェクトの種類によって以下のように分けられる
- particle
- flip-book(sprite)
- Mesh VFX(3dメッシュをエフェクトとして使用)
- ribbon /trail (動くものの軌跡を描画する手法)
- shader-based vfx (shaderによってテクスチャやメッシュの外見をリアルタイムで変化)
- post-processing
- Decal
- Simulation

など．




#### UIは
UIは以下のように構成される
- Sprite / Image: 基本となる矩形描画 9-slice (端を伸ばさない伸縮の対応が必要)
- Text: フォントレンダリング
  - Bitmap Font : スプライトと同様
  - SDF Font(Signed Distance Field): 拡大してもボヤけない高品質テキスト
- Procedural Geometry: 丸角ボタンや円形プログレスバーなど、シェーダーで形状を生成するもの
- Screen Space vs World Space: 画面に張り付くUIか、ゲーム内看板のように3D空間に配置されるUIか


#### デバッグ
デバッグ
- Primitive Shapes: 線(line), ワイヤーフレームの箱(Box),球(Sphere).物理演算の衝突判定などに使用
- Text Overlay: 画面上の特定オブジェクトの頭上に情報を表示
- Gizmous: エディタ的な操作(移動・回転・拡縮)のための軸表示
- Visualizer: 
  - Normal / Tangent 可視化: 法線方向の確認
  - Buffer Visualization: Depth, Roughness, Normal などの中間バッファを画面に並べる機能


### 描画パスによる分類
「不透明」と「半透明」の分離
- Renderパスにおいて、不透明か半透明化によってパスを分けることによって、データ駆動でその制御を可能に
- シャドウパスによる分類(影フラグが立っているものだけを抽出し、影専用の軽量パスに流す)

このように、「どういう手順で描画すうるか(パス)」をデータ定義によって制御可能にできる構造を確保する．

#### A: ビルド時にパスごとの集合を作るか B:毎フレームカリング済みリストを走査して振り分けるか
A
- メリット: 実行時の振り分けコストがほぼゼロ
- デメリット: 動的な変化に弱い．例えば、普段は不透明だが、ダメージを受けたときだけ半透明になるという処理が、ビルド済み集合を別の集合へ移動させるコストが発生する

B
- メリット: 柔軟性が高い．マテリアル状態が毎フレーム変わっても、フラグ判定をするだけだから破綻しない
- デメリット: CPU走査コストがかかる
- 現代の潮流: B
  - 数千~数万個のIDをビット演算でチェックするコストは現代CPUにとっては軽微だから(SIMD並列化と相性が良い)



### 光源の扱い
ライトは以下のような種類がある
- Directional Light
- Point Light
- Spot Light
- Enviroment(Skybox/IBL)

build
1. 全アクターから「Light Component」を抽出
2. 「実行系ライトID」を発行し、ライトのデータだけを連続した配列(LightBuffer)にパックする
3. カメラの視界に入っていないライト(影響範囲が画面外のライト)をこの時点で除外する

実行フェーズ
- パックされたライト配列をUBO(Uniform Buffer Object)やSSBOという仕組みで、1回の手順でGPUに送る


Deffered Rendering
- 通常の描画: メッシュを描画する「ついで」にライトの計算をする
- Deffered: まず色や法線の情報を画面いっぱいに書き出し、その後に「ライトの形をした見えないメッシュ(球体など)」を描画して、  
その範囲内のピクセルにだけ光の計算を適用する



### マテリアルバリエーション
マテリアルの4要素
- シェーダープログラム(Shader Logic): 頂点シェーダーやフラグメントシェーダーのバイナリ(またはソース)
- レンダーステート(render states): 深度テストの有無、ブレンドモード(不透明／加算など)、カリング設定
- テクスチャ・リソース(Texutres/Buffers): カラーマップ、法線マップ、粗さ(Roughness)マップなど
- パラメータ定義(Parameters/Uniforms): 「色を2倍にする」「透明度を0.5にする」といった、数値やベクトルデータ

#### 視点
「同じモデル」について「違うマテリアル」を適用させるよう実装をどう扱うか
- MaterialIDs: ビルド時にリソースIDとは別に「どのPipeline State Object(PSO)を使うか」をID化し、  
同じPSOを持つ実行系ID同士をソートできるようにすると、ドローコールを最適化できる

すぐに実装することは考えないが、これを実装できる構造的余地を残すことを目指す．


#### Gemini
「ID=インデックス」という設計において最も重要なのは、「描画順序の決定ロジック」の中にマテリアルを組み込む余地を作ること
- 今できること: 実行系データ(コマンドパケット)の中に、将来的にマテリアルIDを格納するための`uint32_t materialID`フィールドを用意する
- なぜ重要か: 同じメッシュIDでもマテリアルが違えば、描画の「バケツ」を分ける必要がある．  
逆に、メッシュが違ってもマテリアル(シェーダー)が同じなら、ステート変更を最小限に抑えられる．

マテリアルの「親子関係(インスタンス)」の考慮  
マテリアルは通常、「シェーダー(プログラム)」と「パラメータ(テクスチャや色)」の組合わせ
- 構造的余地: 「マテリアル定義」を以下の2段階でとらえられるようにしておく
- Instance: テンプレートに対して、具体的なテクスチャIDや色の数値を流し込んだもの

- 実装のヒント:ビルド時に「このアクターはどのインスタンスを使うか」を解決し、実行IDにはその「インスタンスID」を紐づける

パラメータ転送の「共通パイプ」を確保する  
マテリアルのバリエーションが増えると、オブジェクトごとに`glUniform`で値を送るのがボトルネックになる
- 構造的余地: 実行系データの集約(ビルド)の際に、「各マテリアルのパラメータを巨大な一塊のメモリ(Uniform Bufferなど)に書き出す」固定を追加できるように設計する  
- 将来のビジョン: 「アクターID 0番はマテリアルパラメータ配列の0番を参照する」といった形にして、GPU側に一括転送(Persistent Mappingなど)できるようにしておくと、  
バリエーションが数千種類になっても速度が落ちない


マテリアルバリエーションに対応するために、ビルドシステムとID管理の中に「マスター」と「インスタンス」の改装を想定しておく

マスターマテリアル
- 内容: シェーダープログラムとレンダーステートの組み合わせ
- 特徴: 切り替えコストが非常に高い
- 設計への影響: これを`ShaderID`と呼び、実行系IDのソートの最優先順位にする　

マテリアルインスタンス
- 内容: マスターマテリアルを使いつつ「テクスチャID」や「色パラーメータ」だけを個別に設定したもの
- 特徴: 切り替えるコストが中程度
- 設計への影響: これを`MaterialID`と呼び、`ShaderID`が同じものの中でさらにソートするために使用する

さらにインスタンスデータとして、個々のパーティクルなどの現在位置、現在の寿命などと階層化できる

### 空間管理
画面外のオブジェクトをどのように管理するか．またそのシステムと紐づけるか
- ビルドされた実行系データがどのような空間構造に配置されるか(Octree, BVH, グリッドなど)
- 「実行系ID」の中に「バウンディングボリューム(AABB)」のデータをパックしておくと、カリング(描画判定)と描画登録がスムーズに


### UIについて
UIは2Dであるだけでなく、ヒエラルキー(親子関係)を厳密に管理し描画順を制御しなければいけない．  

「ビルド->ID化->パケット生成」という共通のパイプの中に、UI専用の「専用レーン」が必要になる

UI専用のID体系とビルドパスを独立して持たせることにする．

3Dアクター/VFX系: `ActorID`配列
- 集約: ワールド行列、AABBの更新
- 抽出: カリング
- パケット化: 3D描画パケット生成
- 最終実行: 共通のレンダリングパイプライン(GL命令発行)

UI系: `UI_ElementID`配列
- 集約: レイアウト計算、親子座標の確定
- 抽出: 基本は全表示
- パケット化: UI描画パケット生成
- 最終実行: 共通のレンダリングパイプライン


### デバッグ描画について
デバッグ描画では、  
「今この瞬間だけ、特定のベクトル方向の線を引きたい」というとき、わざわざアクターをビルドしてID発行するのは面倒
- 解決策: デバッグ系では、IDを通さない「即時モード」の専用レーンを用意する

デバッグシステムの設計
- 簡易命令: `Debug::DrawLine(start, end, color)`のような関数をどこからでも呼べるようにする
- 一時バッファ: 呼ばれた瞬間に動的な頂点配列(std::vectorなど)にデータを放り込む
- 一括クリア: フレームの最後に、その配列を全部描画して中身をからにする

- デバッグビュー（Depth/Normal/Roughness表示）
