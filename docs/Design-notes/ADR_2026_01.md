## 素材データの定義とその集約による意味の生成の区別

### メタ情報
- **ID**: ADR-20260107-01
- **Status**: Proposed
- **Date**: 2026-01-07
- **Owner**: @YM
- **Tags**: Architecture / Engine / App
- **Component**: Asset /  
- **Related**: 

### Context (必須・1行)
*なぜ今これが必要か（1〜2行）*

- データ駆動アーキテクチャを徹底し、システムの柔軟性と処理効率を向上するため
- ゲーム内の意味(のあるまとまりとしてのオブジェクト)を、  
アセットやデータセットを集約(合成)して生成するという構造を成立させるため


### Decision (必須・1行)
*何をどう決めたか（1〜2行）※「する/しない」を明示*

- `Engine`側: 
  - データ(`Asset`が主)リソースのCRUDおよび正規化機能を提供
- `GameApp(データ登録系)`: 
  - Engine側機能を使用して、実データ(リソース)の登録と定義を行う                (アセットの登録)
  - 登録した実データと対応するIDを変数(データ)として保持する．                  (アセットのID化)
  - 素材データを集約することによってゲームとして意味(のあるオブジェクト)を構成する(アセットの合成)
    - この集約はゲーム側で生成した(またはする予定の)IDを集約し、データ化することによって整合的に行う

- `GameApp(データ処理系)`:
  -  各データに対して定義する動作の実装
  -  (要件等: 振舞としての処理もデータ化できる)

#### 実装例: Skill

Skill:   
ゲーム内オブジェクトとして攻撃の単位となる  
以下のデータによって構成される  
- HitBox(論理的当たり判定データ)
- Animation(時間変位を表現する描画データ)
- Sound(音声データ)
- VFX (Animation表現を補助する描画データ)

Skill定義は**事前定義された**これらのデータを  
**イベント時空間**にマッピングするように定義する  

この定義は`SkillBindingDTO`によって行われる

**イベント時空間**とは**FSMエンジン**により定義されたイベント制御の構造のこと(別記)

### Context (必須・1行)

### Decision (必須・1行)
*何をどう決めたか（1〜2行）※「する/しない」を明示*

VFXのデータを以下のように分けて運用する

- VFX定義データ(`VFXDef`): `path/default`などのVFXの定義情報を持つ
- VFXレシピデータ(`VFXRecipe`): 使用側が、`色/サイズ/発生数/速度/重力/ライフ/ランダムシード`などの時間・空間的発生と終了に必要な情報を持つ
- VFX実体データ(`VFXInstanc`): `VFXDef` x `VFXRecipe`によって実際にトリガーされた`VFXInstance`が実行系のデータとなる

これは、SFXやHitboxと同じ構造である

## Audio の設計と実装について
- **ID**: ADR-20260106-01

- **Date**: 2026-01-06
- **Owner**: @YM

- **Component**: `Game/Audio/`API + `Engine/Audio/`APIの責務分離
- **Related**:

### Context (必須)
*なぜ今これが必要か（1〜2行）*

- 共通基盤となるEngine側Auidio実装と、ゲーム特有仕様となるGame側Audio実装の責務の境界をはっきりさせるため

> 決定についてあいまいさが生じた場合、これを確認し、必要であれば更新する


### Decision 
#### `Engine/Audio`API(以降Engine側)責務
- `AudioCmdBuffer`:
  - `AudioSystem`でSoundを再生するために、イベント情報をバッファする 
- `AudioCatalog`: 
  - `SoundDef`を実リソース情報として保持 
  - `SoundID`(正規化ID)によってCRUDの効率化
- `AudioSystem`: 
  - `AudioCatalog`の情報と`AudioCmdBuffer`を利用して、実際にSoundを再生する(実装は`AudioBackend`)


#### `Game/Audio`API(以降Game側)責務
- `SoundKey`:
  - 現状は、**スタブ**
  - 責務は、`AudioCatalog`をO(1)で解決するために、`SoundID(リソース固有ID)`を`SoundKey`としてGame側で表現すること
- `AudioCatalogBuilder`: 
  - Engine側の`AudioCatalog`のCRUDを利用し、実データを`AudioCatalog`へ登録
  - `SoundKey`を作成する



```text
- Engine/Audio
  - Public
    - AudioCatalog
    - AudioSystem

- Game/Audio
  - Public
    - AudioCatalogBuilder
    - AudioIds(SoundKey,SoundID)
```
